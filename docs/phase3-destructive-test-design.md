# 丸投げシステム (Maru-nage v2) フェーズ3 破壊的テスト設計書

## 0. 目的

本書は、フェーズ3のコンテナ認証・起動導線設計を論理的に破壊し、RED テストで保証すべき異常系と合格基準を定義する。

## 1. テスト方針

1. 正常系の裏返しではなく、設計の暗黙前提が崩れたときにどのように安全に失敗するかを定義する。
2. 失敗は可能な限りホスト側起動導線で検知し、コンテナ起動前に止める。
3. 認証情報の漏洩、不要なマウント、secret と環境変数の二重運用を重点的に検証する。

## 2. 破壊シナリオと合格基準

### DT-01: `gh` 未導入

- 攻撃シナリオ: ホスト環境に `gh` コマンドが存在しない。
- Red 条件: `docker compose` 実行に進み、コンテナ起動後に不完全な失敗となる。
- Green 条件: 起動導線が事前に `gh` 不在を検知し、理由を表示して終了する。

### DT-02: `gh auth token` 失敗

- 攻撃シナリオ: `gh auth token` が未ログイン等で非 0 終了する。
- Red 条件: 空または未定義の `GITHUB_TOKEN` で compose 実行へ進む。
- Green 条件: 起動導線が compose 実行前に停止し、再ログインを促す。

### DT-03: 空トークン注入

- 攻撃シナリオ: `gh auth token` が空文字列または空白のみを返す。
- Red 条件: 空の `GITHUB_TOKEN` がコンテナへ渡される。
- Green 条件: 起動導線が空トークンを拒否し、コンテナを起動しない。

### DT-04: `.copilot` マウント残存

- 攻撃シナリオ: compose 定義に `.copilot` のホストマウントが残っている。
- Red 条件: ホストの Copilot 設定ディレクトリがコンテナ内へ露出する。
- Green 条件: compose 定義から該当マウントが除去され、起動に不要となる。

### DT-05: `github_token` secret 残存

- 攻撃シナリオ: `github_token` secret と `GITHUB_TOKEN_FILE` が設定に残っている。
- Red 条件: secret ファイルと環境変数の二重運用が発生する。
- Green 条件: `GITHUB_TOKEN` 環境変数のみを認証経路として使用する。

### DT-06: DB 不一致

- 攻撃シナリオ: DB secret は正しいが、起動導線の改修によって他サービスの既存 DB 起動要件が壊れる。
- Red 条件: GitHub 認証改修が DB 接続失敗やサービス初期化失敗を誘発する。
- Green 条件: DB secret 運用は維持され、GitHub 認証改修が DB 初期化ロジックへ副作用を与えない。

### DT-07: MCP 接続失敗

- 攻撃シナリオ: GitHub 連携先や外部 MCP 依存が利用不能な状態で起動する。
- Red 条件: 認証経路の変更により失敗原因が不明確になり、起動後に曖昧な障害となる。
- Green 条件: 少なくとも認証経路由来の失敗はホスト側で切り分けられ、MCP 側障害と混同しない。

### DT-08: 巨大ファイルまたは過大入力

- 攻撃シナリオ: 起動時や運用手順確認時に巨大ファイル処理や大きな入力を伴うタスクが存在する。
- Red 条件: 認証改修が無関係な大容量処理経路へ影響し、エラーハンドリングが崩れる。
- Green 条件: 認証改修は起動導線と compose 設定に限定され、大容量処理経路の既存制御を壊さない。

## 3. RED で確認すべき項目

1. `docker-compose.prod.yml` に `.copilot` マウントが残っていないこと。
2. `docker-compose.prod.yml` に `github_token` secret と `GITHUB_TOKEN_FILE` が残っていないこと。
3. ホスト側起動導線が `gh` 未導入・未ログイン・空トークンで失敗すること。
4. `brain` と `guardian` が `GITHUB_TOKEN` を必須として扱うこと。
5. DB 用 secret 運用には回帰がないこと。
