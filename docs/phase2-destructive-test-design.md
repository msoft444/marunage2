# 丸投げシステム (Maru-nage v2) フェーズ2 破壊的テスト設計書

## 0. 本書の目的と方針

本書はフェーズ1基本設計書の論理的脆弱性を網羅的に特定し、フェーズ3（テスト実装）で Red 状態から開始するための合格基準を定義する。

### 0.1 破壊の哲学

本書が想定する攻撃者は「設計を正確に読んだ上で、仕様の隙間を突く内部 AI」である。正常系の裏返しではなく、設計者が暗黙に前提とした条件が崩れるシナリオを優先する。

### 0.2 テスト項目の読み方

各テスト項目は以下の構造で記述する。

- **ID:** カテゴリ-連番
- **攻撃シナリオ:** 設計が崩壊する具体的状況
- **Red 条件:** テストが失敗（Red）となる状態。実装が存在しない初期状態ではこの条件が成立する
- **Green 条件:** 防御が機能し、テストが成功（Green）と見なされる論理条件
- **検証対象:** テストが主に検査する設計要素

## 1. 自己改修のデッドロックと状態不整合

フェーズ1設計 §4.5 は自己改修を隔離ワークスペースで行い、本番反映を Guardian に委任する。この委任チェーンが途切れるシナリオを検証する。

### DL-01: promote 発行元 Brain の消失

- **攻撃シナリオ:** 作業 AI が `promote_release` タスクを発行した直後、Guardian が blue-green 切替を開始し、旧 Brain コンテナを停止する。旧 Brain 上で動作していた管理 AI のタスクは `running` のまま残り、`lease_expires_at` の更新が止まる。新 Brain 起動後、Guardian はゾンビ判定で当該タスクを再取得対象にするが、再取得すべきか完了済みとすべきかの判断基準が存在しない。
- **Red 条件:** `promote_release` タスクが `succeeded` になった後、発行元タスク（親タスク）が `running` のまま永続する。
- **Green 条件:** Guardian が `promote_release` を完了した時点で、発行元タスクの `lease_owner` が消失済みであれば、当該タスクを `succeeded`（promote 結果に連動）または `blocked`（promote 失敗時）へ強制遷移し、`logs` に `orphan_task_resolved` イベントを記録する。
- **検証対象:** §1.4 タスク取得と同期方式、§4.5 再帰的自己改修シナリオ

### DL-02: Guardian 自身の更新パラドックス

- **攻撃シナリオ:** 対象リポジトリがシステム自身であり、変更対象に Guardian の Dockerfile と起動スクリプトが含まれる。Guardian は「聖域」だが、自身を更新するタスクを自身が実行する必要がある。旧 Guardian を停止した瞬間、新 Guardian のヘルス確認を行う主体が消滅する。
- **Red 条件:** Guardian 更新中にシステム全体の監視と復旧機能が不在になる時間帯が存在する。
- **Green 条件:** Guardian の自己更新は以下の手順を強制する。(1) 新 Guardian を別名・別ポートで起動し、ヘルスチェックに合格させる。(2) 新 Guardian が旧 Guardian の監視ロールを引き継ぐ。(3) 旧 Guardian を停止する。引き継ぎの全段階を `logs` に記録し、新 Guardian のヘルスチェックが失敗した場合は旧 Guardian を維持して `blocked` とする。
- **検証対象:** §4.5、§7.5、§8 可用性

### DL-03: DB スキーマ変更を含む自己改修

- **攻撃シナリオ:** 自己改修が `tasks` テーブルに新しい列を追加するマイグレーションを含む。隔離環境でテスト成功後、Guardian が本番 DB にマイグレーションを適用する。適用の瞬間、稼働中の Brain と Librarian は未知の列を持つスキーマ上で動作し、INSERT や SELECT が予期せぬ挙動を示す。
- **Red 条件:** マイグレーション適用直後に、旧コード上の Brain が DB 操作でエラーを起こしタスクが失敗する。
- **Green 条件:** DB スキーマ変更を含む自己改修は blue-green の切替順序を「(1) 新コード側サービス群を起動、(2) 新 DB スキーマ適用（後方互換な追加のみ許可、列削除は別タスクで遅延実行）、(3) 旧コード側を停止」とし、後方互換でないスキーマ変更は `blocked` として人間承認を要求する。
- **検証対象:** §4.5、§1.1

### DL-04: promote_release と restart_service の同時発行

- **攻撃シナリオ:** Dashboard から `restart_service` が発行され、同時に作業 AI が `promote_release` を発行する。Guardian は両方を取得する。restart が先に実行されると、promote 対象コンテナが消滅し promote が失敗する。promote が先だと、restart 対象が新系に切り替わり、意図しないコンテナが再起動される。
- **Red 条件:** 両タスクが同一サービスを対象とする場合に、両方とも `succeeded` になるが実際の状態が不定になる。
- **Green 条件:** Guardian は同一サービスを対象とする `promote_release` と `restart_service` を排他制御する。先行タスク実行中は後続を `blocked` にし、先行完了後に後続の前提条件を再評価する。
- **検証対象:** §1.4、§4.5、§2.2 エージェント再起動 UI

### DL-05: 自己改修中のハートビート断絶による誤ゾンビ判定

- **攻撃シナリオ:** Brain が自己改修の大規模テスト実行中に、Copilot CLI の推論が `lease_expires_at` を超える長時間を要する。Guardian がゾンビと判定し、タスクを強制的に `failed` にして再取得対象にする。Brain は推論完了後に結果を書き込もうとするが、タスクは既に別インスタンスに再リースされている。
- **Red 条件:** 同一タスクに対して2つの作業 AI が並行して結果を書き込み、`messages` と成果物が破損する。
- **Green 条件:** タスクへの結果書き込み時に `lease_owner` の一致を確認する。不一致の場合は書き込みを拒否し、`logs` に `stale_lease_write_rejected` を記録する。推論が長時間になる場合はハートビート更新を推論ループ内で定期実行する。
- **検証対象:** §1.4、§8 可用性

## 2. リソース競合とポート衝突

フェーズ1設計 §4.3 はポート予約を DB ロックとリトライで管理するが、DB ロックと実際のポート状態の間にはタイムギャップが存在する。

### RC-01: TOCTOU 競合によるポート二重取得

- **攻撃シナリオ:** 管理 AI-A がポートアロケータ行をロックし、ポート 18080 の空きを Docker と OS の両方で確認し、DB にポート予約を記録してロックを解放する。その直後、管理 AI-A がコンテナを起動するまでの間に、ホスト上の別プロセス（marunage 外）が同ポートを bind する。コンテナ起動が `port already in use` で失敗する。
- **Red 条件:** DB 上はポート予約済みだがコンテナ起動に失敗し、タスクが indefinite な状態に留まる。
- **Green 条件:** コンテナ起動失敗時に、DB 上の予約を解放し、次候補ポートで再試行する。再試行は §4.3 の最大 8 回ルールに含める。起動成功を確認するまでポート予約を「仮」状態とし、確認後に「確定」へ遷移する。
- **検証対象:** §4.3 動的ポート割当

### RC-02: ポートレンジ枯渇

- **攻撃シナリオ:** 同時に 100 件以上のタスクが発行され、Dashboard 用レンジ 18080-18179 の全 100 ポートが予約済みとなる。101 件目のタスクは 8 回リトライしても空きがなく `blocked` になるが、先行タスクの大半はまだ `running` であり解放されない。
- **Red 条件:** レンジ枯渇後に `blocked` タスクが蓄積し続け、システムが機能停止する。
- **Green 条件:** (1) ポートレンジ枯渇時に Guardian へ `port_exhaustion` アラートを送出する。(2) `finished_at` が記録済みだがポートが解放されていない孤立予約を Guardian が定期回収する。(3) レンジは設定で拡張可能とし、拡張時に既存予約と衝突しないことを検証する。
- **検証対象:** §4.3、§8 可用性

### RC-03: root_task_id ハッシュ衝突による初回候補の同一化

- **攻撃シナリオ:** 2つの異なる `root_task_id` が同一のハッシュ初回候補ポートを算出する。両者がほぼ同時にアロケータ行をロックしに行く場合、先行者はロックを取得して予約するが、後続者はロック待ち後に同一ポートを候補として算出し衝突を検知する。設計通りなら次候補へ進むが、ハッシュ関数の分布が偏っていると連続衝突が発生する。
- **Red 条件:** ハッシュ衝突が連鎖し、8 回のリトライ上限内で空きポートに到達できない。
- **Green 条件:** 衝突検知時の次候補選定は線形走査（ハッシュ + 1, +2, ...）ではなく、リトライ回数をシードに加えた再ハッシュまたはランダムオフセットとし、同一ポートへの収束を防ぐ。
- **検証対象:** §4.3

### RC-04: コンテナ名衝突（外部起源）

- **攻撃シナリオ:** ホスト上に marunage 管理外のコンテナが `mn2-` プレフィクスと同名規則のコンテナ名で稼働している。管理 AI が生成したコンテナ名が偶然一致し、`docker run` が `Conflict` エラーを返す。
- **Red 条件:** コンテナ名衝突がポートリトライルートに含まれず、即座にタスクが `failed` になる。
- **Green 条件:** コンテナ起動前に `docker inspect` で同名コンテナの存在を確認する。存在する場合は `short_hash` を再生成して最大 3 回リトライする。それでも衝突する場合のみ `blocked` とする。
- **検証対象:** §4.2、§4.3

### RC-05: ネットワーク名の衝突と残存

- **攻撃シナリオ:** タスク完了後、`mn2-net-{root_task_id}` ネットワークの削除に失敗する（接続中コンテナが残存）。同じ `root_task_id` でリトライされた場合、既存ネットワークの再利用が意図されていないのに再利用され、旧コンテナと新コンテナが同一ネットワーク上で混在する。
- **Red 条件:** リトライタスクが旧タスクのゾンビコンテナと同一ネットワーク上で通信し、予測不能な挙動を示す。
- **Green 条件:** タスク開始時に同名ネットワークの存在を確認し、接続コンテナがあれば全停止してからネットワークを削除する。削除失敗時は別名ネットワークを作成し、`runtime_spec_json` を更新する。
- **検証対象:** §4.2

## 3. 機微情報の多重突破

フェーズ1設計 §6.2 のプレコミットスキャンは「1段の難読化解除」「変数名文脈検査」「高エントロピー検査」を行うが、これらを迂回する経路を検証する。

### SS-01: 多段エンコード（2段以上）

- **攻撃シナリオ:** 作業 AI が GitHub token を `base64(base64(token))` で二重エンコードし、変数名を `ENCODED_DATA` としてソースコードに埋め込む。§6.2 は「難読化解除を一段だけ試す」と明記されているため、二段目のデコード後に出現するトークンパターンを検知できない。
- **Red 条件:** 二重エンコードされたトークンがコミットをすり抜ける。
- **Green 条件:** 難読化解除は再帰的に適用し、上限を 3 段とする。各段で既知パターンとエントロピー判定を実施する。3 段を超えるネスト自体を異常とみなし、`WARN` ログを記録して人間確認を要求する。
- **検証対象:** §6.2 ステップ 5

### SS-02: Unicode ホモグリフによるパターン回避

- **攻撃シナリオ:** 変数名の `PASSWORD` を `ΡASSWORD`（先頭がギリシャ文字 Ρ）に置換する。§6.2 ステップ 6 の文字列マッチは ASCII を前提とするため、変数名文脈検査をすり抜ける。値には平文パスワードが直書きされる。
- **Red 条件:** ホモグリフ変数名に代入された機微情報がコミットをすり抜ける。
- **Green 条件:** 変数名文脈検査を Unicode 正規化（NFKD）後に適用する。加えて、ソースコード内の非 ASCII 識別子の使用自体を `WARN` として検出し、機微情報キーワードとの類似度を判定する。
- **検証対象:** §6.2 ステップ 6

### SS-03: 機微情報の複数ファイル分割

- **攻撃シナリオ:** トークン `ghp_abcdef123456` を `file_a.py` に `PART1 = "ghp_abcdef"` 、`file_b.py` に `PART2 = "123456"` と分割し、実行時に結合する。個々のファイルではパターンマッチもエントロピー閾値も超えない。
- **Red 条件:** 分割されたトークン断片がそれぞれ単独ファイルのスキャンをすり抜ける。
- **Green 条件:** (1) 既知プレフィクスパターン（`ghp_`、`gho_`、`Bearer `、`ey` 等）を含む文字列は長さに関係なく検出対象とする。(2) 同一コミット内の全ステージファイルを結合したクロスファイル検査を実施する。(3) `PART` `FRAGMENT` `CHUNK` `HALF` `SEGMENT` 等の分割を示唆する変数名パターンを `WARN` として検出する。
- **検証対象:** §6.2 ステップ 3, 6

### SS-04: バイナリファイルへの埋め込み

- **攻撃シナリオ:** 作業 AI がテスト用フィクスチャとして画像ファイルを生成し、その EXIF メタデータや末尾バイトにトークンを埋め込む。§6.2 は「差分本文を対象に」スキャンすると記述しているが、バイナリ差分のテキスト表現では埋め込みが見えない場合がある。
- **Red 条件:** バイナリファイル内に埋め込まれた機微情報がスキャンをすり抜ける。
- **Green 条件:** (1) バイナリファイルのコミットは原則ホワイトリスト方式とし、許可拡張子以外の追加を `WARN` で通知する。(2) 許可されたバイナリでも、raw バイト列に対してパターンマッチを実行する。(3) ファイルサイズとエントロピーが閾値を超えるバイナリは人間確認を要求する。
- **検証対象:** §6.2 ステップ 3

### SS-05: git commit --amend / rebase による事後挿入

- **攻撃シナリオ:** 作業 AI のコミット前スキャンを正常に通過したコミットに対し、後続の作業 AI が `git commit --amend` で機微情報を含むファイルを追加する。amend はスキャン対象の「新規コミット」ではなく「既存コミットの修正」であるため、スキャンフローの起動条件に合致しない可能性がある。
- **Red 条件:** amend または rebase によって追加されたファイルがスキャンされずにコミット履歴に残る。
- **Green 条件:** スキャンフックは `git commit` だけでなく `git commit --amend`、`git rebase`、`git cherry-pick`、`git merge` のすべてで起動する。起動条件はコマンド名ではなく、HEAD の変更を検知するポストフックとし、変更後の差分全体を再スキャンする。
- **検証対象:** §6.2

### SS-06: コミット後の参照リセットによる残存

- **攻撃シナリオ:** 作業 AI がクリーンなコミットを作成した後、`git update-ref` で別のコミット（機微情報を含む）を作業ブランチの HEAD に差し替える。プレコミットスキャンは通過済みの履歴に基づいて動作するため、参照操作後の HEAD は検査されない。
- **Red 条件:** 参照操作により、スキャン済みでないコミットがプッシュ対象ブランチの HEAD になる。
- **Green 条件:** プッシュ前にもスキャンを実施する。プッシュ対象のコミット範囲（ローカルのみのコミット全体）に対して、各コミットの差分を再スキャンする。`git push` の実行も作業 AI の管理下に置き、直接実行を禁止する。
- **検証対象:** §6.2、§6.3 提出防御

### SS-07: エントロピー検査の偽陽性による業務妨害

- **攻撃シナリオ:** 正当な SHA-256 ハッシュ、UUID、暗号化テスト用フィクスチャ、圧縮済みデータを含むコミットに対し、エントロピー検査が偽陽性を出し続ける。作業 AI はコミットできず、タスクが `failed` と `retry` のループに陥る。
- **Red 条件:** 正当な高エントロピー文字列を含むコミットが永続的にブロックされ、タスクが完了不能になる。
- **Green 条件:** (1) ホワイトリスト機構を設け、既知の安全なパターン（SHA-256 の 64 文字 hex、UUID v4、テストフィクスチャマーカー付き領域）をスキャン対象から除外する。(2) 同一ファイルの同一行で 3 回以上偽陽性が発生した場合、`messages` 経由で人間に判断を委ね、ループを停止する。
- **検証対象:** §6.2 ステップ 4、§6.3

## 4. 隔離環境の脱獄

フェーズ1設計 §4.1 は `/workspace/{root_task_id}/` を隔離境界とするが、境界の強制機構が明示されていない。

### JB-01: シンボリックリンクによる脱出

- **攻撃シナリオ:** 作業 AI が `/workspace/{task_id}/repo/` 内に `/etc/passwd` や `/system_docs/` を指すシンボリックリンクを作成する。後続のファイル読み書き操作がリンクを辿り、隔離境界外のファイルを読み取りまたは上書きする。
- **Red 条件:** ワークスペース内のシンボリックリンク経由で、`/workspace/{task_id}/` 外部のファイル内容を取得または変更できる。
- **Green 条件:** (1) ファイル操作前にパスを `realpath` で正規化し、正規化後のパスが `/workspace/{task_id}/` のプレフィクスを持つことを検証する。(2) シンボリックリンクの作成自体を監視し、リンク先が境界外を指す場合はブロックして `logs` に `sandbox_escape_blocked` を記録する。
- **検証対象:** §4.1、§5.3

### JB-02: パストラバーサルによる脱出

- **攻撃シナリオ:** 作業 AI がファイルパスに `../../` を含むパスを指定する。DB の `workspace_path` は正しいが、Copilot CLI や Git 操作に渡すファイルパスの検証が不十分で、相対パスが隔離境界を突破する。
- **Red 条件:** `../` を含むパス指定により `/workspace/{task_id}/` 外部のファイルを操作できる。
- **Green 条件:** すべてのファイルパス引数は「正規化 → プレフィクス検証 → 許可」の順で処理する。正規化前の文字列に `..` を含むパスは即座に拒否する。
- **検証対象:** §4.1、§5.4

### JB-03: Docker ボリュームマウントによるホスト侵食

- **攻撃シナリオ:** 作業 AI が Brain コンテナ内の Docker CLI を使って新しいコンテナを起動する際、`-v /:/host_root` のようにホストルートをマウントする。DooD（Docker outside of Docker）設計により、Brain は Docker CLI でホスト上の任意操作が可能である。
- **Red 条件:** 作業 AI が起動したコンテナがホストの任意ディレクトリへの読み書きアクセスを持つ。
- **Green 条件:** (1) Docker CLI の実行はラッパー経由に限定し、`-v` / `--mount` 引数のソースパスを `/workspace/{task_id}/` 配下と明示的に許可されたパスに制限する。(2) `/var/run/docker.sock`、`/etc`、`/root`、`/home` のマウントを拒否する。(3) 許可リスト外のマウントは `logs` に `docker_mount_blocked` として記録する。
- **検証対象:** §4.4、§7.2

### JB-04: Docker CLI による特権コンテナ起動

- **攻撃シナリオ:** 作業 AI が `docker run --privileged` または `--cap-add=ALL` でコンテナを起動し、cgroup 脱出や `/proc` 経由でホスト OS を操作する。
- **Red 条件:** 作業 AI が起動したコンテナが特権モードで稼働する。
- **Green 条件:** Docker CLI ラッパーが `--privileged`、`--cap-add`、`--pid=host`、`--network=host`、`--ipc=host` を含むフラグを検出した場合、実行をブロックする。許可されるケイパビリティを明示的にホワイトリスト化する。
- **検証対象:** §4.4、§8 安全性

### JB-05: workspace_path カラムの汚染

- **攻撃シナリオ:** 管理 AI が `tasks.workspace_path` に `/workspace/../etc` のようなトラバーサルパスを書き込む。後続の作業 AI はこの値を信頼して操作対象ディレクトリとする。
- **Red 条件:** `workspace_path` に格納されたパスが `/workspace/` 配下でない場合に、作業 AI がそのパスで操作を開始する。
- **Green 条件:** DB アクセッサーの書き込み時と読み取り時の両方で `workspace_path` を正規化し、`/workspace/` プレフィクスを検証する。検証失敗時はタスクを `blocked` にする。
- **検証対象:** §1.1、§4.1

### JB-06: Git サブモジュールによる外部パス参照

- **攻撃シナリオ:** 対象リポジトリの `.gitmodules` に `path = ../../outside` のような定義がある場合、`git submodule update` が隔離境界外にファイルを展開する。
- **Red 条件:** `git submodule update` により `/workspace/{task_id}/` 外にファイルが作成される。
- **Green 条件:** `git submodule update` の前に `.gitmodules` 内の全 `path` エントリを検証し、正規化後のパスがリポジトリルート配下に収まることを確認する。収まらない場合はサブモジュール更新を中止する。
- **検証対象:** §4.1、§5.3

## 5. 外部依存の全損とフェイルセーフ

フェーズ1設計はMariaDB を唯一の通信ハブとし、ChromaDB、GitHub MCP、docker.sock を外部依存として持つ。これらが利用不可になった場合の挙動を検証する。

### EX-01: MariaDB 完全停止

- **攻撃シナリオ:** MariaDB コンテナが異常終了する。すべてのサービスが通信ハブを失い、タスク取得、ステータス更新、ログ記録、メッセージ送信が不可能になる。Brain は実行中タスクの結果を書き込めず、Guardian はゾンビ検知もできない。
- **Red 条件:** MariaDB 停止中にすべてのサービスがパニックし、DB 復旧後に状態の整合性が回復しない。
- **Green 条件:** (1) 各サービスは DB 接続喪失を検知した場合、ローカルの WAL（Write-Ahead Log）ファイルに操作を退避する。(2) DB 復旧後に WAL を再生しつつ、タイムスタンプの矛盾を検知して競合解決する。(3) Guardian は MariaDB コンテナの自動再起動を試み、失敗時は Dashboard へのヘルスチェック経路（DB 非依存の最低限の手段）でアラートする。(4) DB 停止中に新規タスクの受付を停止する。
- **検証対象:** §1.4、§8 可用性

### EX-02: ChromaDB 停止中のナレッジ検索

- **攻撃シナリオ:** ナレッジクエリタスクの実行中に ChromaDB が応答不能になる。Librarian は検索結果を返せず、タスクは `running` のまま停滞する。Brain 側の作業 AI は検索結果を待ち続け、フェーズ全体がブロックされる。
- **Red 条件:** ChromaDB 障害がタスクチェーン全体の停止を引き起こす。
- **Green 条件:** (1) ChromaDB への接続にタイムアウトを設定する。(2) タイムアウト時は `messages` に「ナレッジ検索失敗・ChromaDB 不達」を記録し、利用可能な代替情報源（GitHub MCP のみ、またはキャッシュ済みチャンク）でフォールバック検索を実施する。(3) フォールバックも失敗した場合は `knowledge_query` タスクを `failed` にし、呼び出し元タスクに検索不能を通知する。呼び出し元は検索なしで続行するか人間判断を仰ぐか選択する。
- **検証対象:** §3.5、§8 可用性

### EX-03: GitHub MCP 接続障害

- **攻撃シナリオ:** GitHub MCP サーバーへの接続が DNS 障害またはレート制限により不可能になる。サブモジュール登録と実体取得が失敗し、ナレッジ検索の一部が欠損する。
- **Red 条件:** MCP 障害時に Librarian がハングし、他のナレッジ操作（PDF 登録等）も巻き添えで停止する。
- **Green 条件:** (1) GitHub MCP 呼び出しにサーキットブレーカーを設け、連続 N 回失敗でオープン状態にする。(2) オープン中はMCP依存の検索結果を「取得不可」として部分結果を返し、ChromaDB のみの結果で応答する。(3) サブモジュール登録は MCP 復旧後にリトライ可能な `queued` 状態で保持する。(4) Librarian の他の機能（PDF 登録等）は MCP 障害の影響を受けない。
- **検証対象:** §3.4、§3.5

### EX-04: docker.sock 利用不可

- **攻撃シナリオ:** ホスト側でDocker デーモンが再起動中、または `docker.sock` のパーミッションが変更され、Brain コンテナからの Docker CLI 実行が `Cannot connect to the Docker daemon` で全失敗する。作業 AI はコンテナの起動、テスト実行環境の構築ができない。
- **Red 条件:** docker.sock 障害が Brain 内の全タスクを即座に `failed` にし、復旧後の自動再開が行われない。
- **Green 条件:** (1) Docker CLI 実行失敗を一時的エラーと永続的エラーに分類する。接続エラーは一時的とみなし、固定バックオフで最大 5 回リトライする。(2) 永続的失敗時はタスクを `blocked` にし、Docker CLI を必要としないサブタスク（コード生成、ドキュメント作成等）は続行可能とする。(3) Guardian が docker.sock の状態を監視し、復旧を検知したら `blocked` タスクを再評価する。
- **検証対象:** §4.4、§7.2

### EX-05: Copilot CLI 認証失効

- **攻撃シナリオ:** タスク実行中に Copilot CLI の認証トークンが期限切れになる。CLI は認証エラーを返すが、作業 AI はこれを推論失敗と区別できず、無限にリトライする。
- **Red 条件:** 認証失効を推論エラーと誤認し、`max_retry` まで無意味なリトライを繰り返す。
- **Green 条件:** (1) Copilot CLI の終了コードとエラーメッセージを分類し、認証エラーを専用パターンで検知する。(2) 認証エラー検知時は `retry_count` を進めず、タスクを `blocked` にして `logs` に `copilot_auth_expired` を記録する。(3) Guardian または Dashboard 経由で認証更新が完了した旨を通知するまでリトライしない。
- **検証対象:** §2 Brain 内部フロー、§1.5

### EX-06: ホストディスク容量枯渇

- **攻撃シナリオ:** 隔離ワークスペースの大量ファイル生成、ログの蓄積、Docker イメージのキャッシュによりホストのディスクが枯渇する。DB への書き込みが失敗し、コンテナ起動も失敗し、ログすら書けなくなる。
- **Red 条件:** ディスク枯渇がサイレントに進行し、最終的にデータ損失が発生する。
- **Green 条件:** (1) Guardian がホストのディスク使用率を定期監視し、閾値（例: 85%）超過で `WARN`、臨界値（例: 95%）超過で新規タスク受付を停止する。(2) 完了済みタスクの隔離ワークスペースに保持期限を設け、期限超過分を Guardian が自動削除する。(3) Docker の未使用イメージ、停止コンテナ、未使用ネットワークの定期削除を Guardian が実施する。
- **検証対象:** §4.1、§8 可用性

## 6. タスク状態遷移の論理的破壊

フェーズ1設計 §1.4 のタスク取得方式は FOR UPDATE ロックに依存するが、状態遷移の前提条件が完全には定義されていない。

### TS-01: 状態の逆行

- **攻撃シナリオ:** バグまたは悪意により、`succeeded` 状態のタスクが `running` に戻される。テスト結果とアーティファクトは既に参照されており、再実行は矛盾した状態を生む。
- **Red 条件:** 終端状態（`succeeded`、`failed`、`cancelled`）から非終端状態への遷移が成功する。
- **Green 条件:** DB アクセッサーが状態遷移ルールを強制する。許可される遷移は以下のみとする。`queued → leased → running → {succeeded | failed | blocked | waiting_approval}`、`blocked → queued`（再試行時）、`waiting_approval → {succeeded | cancelled}`、`running → cancelled`（強制停止時）。終端状態からの遷移は DB レベルまたはアクセッサーレベルで拒否する。
- **検証対象:** §1.1、§1.4

### TS-02: 二重リースの発生

- **攻撃シナリオ:** Brain の 2 つのワーカースレッドがほぼ同時に同一タスクをリースしようとする。`FOR UPDATE` は同一トランザクション内では排他制御するが、SELECT の結果を確認してから UPDATE するまでの間にタイムアウトやデッドロック検知による自動ロールバックが発生した場合、リトライロジックが同一タスクを再度リースする可能性がある。
- **Red 条件:** 同一タスクに対して異なる `lease_owner` が記録され、2 つのワーカーが同時に実行する。
- **Green 条件:** リース取得は単一の原子的 UPDATE 文（`UPDATE tasks SET status='leased', lease_owner=? WHERE id=? AND status='queued'` の affected rows = 1 で成功判定）とし、SELECT + UPDATE の 2 ステップにしない。
- **検証対象:** §1.4

### TS-03: lease_expires_at 設定値の不適切さ

- **攻撃シナリオ:** リース期限が短すぎる場合（例: 30 秒）、正常な推論処理中にゾンビ判定される。長すぎる場合（例: 24 時間）、実際のゾンビが長時間検知されない。適切な値の基準がフェーズ1設計に記載されていない。
- **Red 条件:** リース期限の設定値がフェーズや処理種別を考慮せず一律であるため、誤ゾンビ判定または検知遅延が発生する。
- **Green 条件:** リース期限はタスク種別とフェーズに応じて動的に設定する。(1) CLI 推論タスクは推論時間の実績統計から上限を設定する（初期値は 15 分、実績に応じて調整）。(2) Docker 操作タスクはコンテナビルド時間を考慮し 30 分とする。(3) ハートビート間隔はリース期限の 1/3 以下とする。(4) `model_contract_json` にリース期限の推奨値を含めることを許可する。
- **検証対象:** §1.4、§1.5

### TS-04: max_retry 到達後の永続 failed 蓄積

- **攻撃シナリオ:** 一時的な外部障害（ネットワーク瞬断等）により複数タスクが `max_retry` に到達して `failed` になる。障害復旧後、これらのタスクは自動的に再試行されず、人間が気づくまで放置される。
- **Red 条件:** `max_retry` 到達の `failed` タスクが自動復旧の対象にならず、依頼全体が黙って停止する。
- **Green 条件:** (1) `failed` タスクが発生した場合、`root_task_id` 単位で影響範囲を `messages` に記録する。(2) Guardian が `failed` タスクの失敗理由を分類し、一時的障害起因のものには `max_retry` リセット付きの再投入を提案する（自動ではなく `waiting_approval` で人間判断）。(3) Dashboard のトップ画面に未解決 `failed` タスク数を常時表示する。
- **検証対象:** §1.1、§1.4、§8 可用性

## 7. モデル契約の論理的破壊

フェーズ1設計 §1.5 のモデル契約機構は、管理 AI と作業 AI 間の信頼に依存している。

### MC-01: contract_digest の算出アルゴリズム未定義

- **攻撃シナリオ:** 管理 AI と作業 AI が `contract_digest` の算出に異なるフィールド順序、JSON シリアライズ形式、ハッシュアルゴリズムを使用する。結果として正当な契約が検証失敗し、すべてのタスクが `blocked` になる。
- **Red 条件:** 正当なモデル契約が検証不一致で `blocked` となる。
- **Green 条件:** `contract_digest` の算出仕様を以下のとおり固定する。(1) 対象フィールドは `phase`, `model`, `cli_profile`, `fallback_allowed`, `contract_version` の5つとし、この順序で結合する。(2) JSON は RFC 8785（JCS: JSON Canonicalization Scheme）に従って正規化する。(3) ハッシュは SHA-256 とし、hex 表現で格納する。(4) 算出ロジックは共通ライブラリとして単一実装を使用する。
- **検証対象:** §1.5

### MC-02: Copilot CLI のモデルエイリアス問題

- **攻撃シナリオ:** `model_contract_json.model` に `gpt-5.4` を指定したが、Copilot CLI が内部的にこれを `gpt-5.4-2026-02` にエイリアス解決する。作業 AI が起動後にログに記録するモデル名は CLI が返す名前であり、契約上の名前と一致しない。監査ログが常に不一致を報告する。
- **Red 条件:** CLI が返すモデル名と契約のモデル名が一致せず、正常な実行が監査違反として記録される。
- **Green 条件:** (1) `model_contract_json` にエイリアス許可リスト（`allowed_aliases`）フィールドを追加し、既知のエイリアス解決結果を事前定義する。(2) 監査ログでは契約名と実際の CLI 解決名の両方を記録し、エイリアス許可リスト内であれば `model_validated`、リスト外であれば `model_alias_mismatch` として検出する。
- **検証対象:** §1.5

### MC-03: model_contract_json の部分破損

- **攻撃シナリオ:** DB の JSON カラムにおいて、書き込み時のトランケーションまたは文字エンコーディングの問題で `model_contract_json` が不正な JSON になる。作業 AI が解析に失敗し、例外で停止する。
- **Red 条件:** 不正な JSON を持つタスクが作業 AI の未ハンドル例外を引き起こす。
- **Green 条件:** (1) `model_contract_json` の読み取り時に JSON パースエラーを捕捉し、タスクを `blocked` にして `logs` に `contract_parse_error` を記録する。(2) 書き込み時に JSON スキーマバリデーションを実施し、必須フィールドの欠落や型不一致を拒否する。
- **検証対象:** §1.5、§1.1

## 8. Librarian の破壊

### LB-01: 悪意ある PDF（zip bomb 相当）

- **攻撃シナリオ:** Dashboard 経由でアップロードされた PDF が、展開時に数 GB のテキストを生成する構造を持つ。`pymupdf` の処理がメモリを枯渇させ、Librarian コンテナが OOM Kill される。
- **Red 条件:** 単一 PDF の処理が Librarian コンテナ全体を停止させ、他のナレッジ操作も巻き添えになる。
- **Green 条件:** (1) PDF 受領時にファイルサイズ上限を設ける。(2) ページ抽出を逐次処理とし、ページあたりの抽出テキストサイズに上限を設定する。(3) 処理全体にタイムアウトを設け、超過時は処理を中止して部分結果を記録する。(4) Librarian コンテナに Docker のメモリ制限を設定し、OOM Kill 時は Guardian が再起動する。
- **検証対象:** §3.3、§7.3

### LB-02: テキスト抽出不能な PDF

- **攻撃シナリオ:** 全ページが画像のみ（スキャン PDF）で、`pymupdf` のテキスト抽出が空文字列を返す。Librarian は空の Markdown を生成し、空のチャンクを ChromaDB に登録する。後の検索で無意味な結果を返す。
- **Red 条件:** テキスト抽出量がゼロまたは極少の PDF がナレッジとして登録され、検索品質を劣化させる。
- **Green 条件:** (1) ページ単位の抽出テキスト量を監視し、全ページの合計が閾値未満の場合は「テキスト抽出不能」として登録を拒否する。(2) 登録拒否時は `messages` に理由を記録し、Dashboard に表示する。(3) 将来の OCR 対応は対象外とし、現時点では拒否が正しい挙動であることを明示する。
- **検証対象:** §3.3

### LB-03: PDF 登録と削除の同時実行

- **攻撃シナリオ:** Dashboard から同一 PDF の登録と削除がほぼ同時に発行される。Librarian は登録処理中に ChromaDB へチャンクを書き込んでいる最中に、削除処理が同一 `knowledge_id` の全チャンクを削除する。結果としてチャンクの一部だけが残存する。
- **Red 条件:** 同一 `knowledge_id` に対する登録と削除の並行実行により、ChromaDB 上に不完全なチャンクが残存する。
- **Green 条件:** (1) `knowledge_id` ごとの操作を排他制御する（`knowledge_id` 単位のロックまたは Librarian 内のシリアライズキュー）。(2) 登録中の `knowledge_id` に対する削除要求は、登録完了後に実行するか、登録を中止してから削除する。
- **検証対象:** §3.2、§3.3

### LB-04: ChromaDB とメタデータ DB の不整合

- **攻撃シナリオ:** PDF 登録のステップ 6（ChromaDB ベクトル登録）が成功した後、ステップ 7（DB メタデータ記録）が失敗する。ChromaDB にはチャンクが存在するが、DB 上は登録されていない状態になる。逆に、DB 記録後に ChromaDB への非同期削除が失敗し、削除済みナレッジのチャンクが検索結果に現れる場合もある。
- **Red 条件:** ChromaDB と MariaDB の間でナレッジの存在状態が不一致になり、ゴーストチャンクまたは孤立メタデータが発生する。
- **Green 条件:** (1) 登録は「DB メタデータを `pending` で先行挿入 → ChromaDB 登録 → DB を `active` に更新」の順とし、失敗時は `pending` のままとする。(2) Guardian が定期的に `pending` 状態のまま一定時間経過したメタデータを検出し、対応する ChromaDB チャンクの有無を確認して状態を修正する。(3) 削除は「DB を `deleting` に更新 → ChromaDB 削除 → DB メタデータを削除」の順とし、ChromaDB 削除失敗時はリトライキューに入れる。
- **検証対象:** §3.3、§3.5

## 9. Dashboard / UI の破壊

### UI-01: content_md 経由の XSS

- **攻撃シナリオ:** AI が生成した `messages.content_md` に、Markdown のインラインHTML として `<script>` タグや `onerror` イベントハンドラを含む画像タグが埋め込まれる。Dashboard の詳細画面がこれをそのままレンダリングし、ブラウザ上で任意スクリプトが実行される。
- **Red 条件:** `content_md` 内の HTML がブラウザ上でスクリプト実行可能な状態でレンダリングされる。
- **Green 条件:** (1) Markdown レンダリング時に HTML サニタイズを適用し、許可タグ（`p`, `h1`-`h6`, `ul`, `ol`, `li`, `code`, `pre`, `table`, `a`, `img`, `em`, `strong`, `blockquote`）のみを通過させる。(2) `a` タグの `href` は `http`/`https` スキームのみ許可する。(3) `img` タグの `src` は内部 URL のみ許可する。(4) イベントハンドラ属性を全除去する。
- **検証対象:** §2.2 開発履歴詳細画面

### UI-02: 大量メッセージによるタイムライン表示の崩壊

- **攻撃シナリオ:** フェーズ4の自律デバッグループが数千回のリトライを行い、各回で `messages` と `logs` を記録する。履歴詳細画面が数万行のタイムラインを一括読み込みし、ブラウザがフリーズする。
- **Red 条件:** メッセージ数が 1 万件を超える `root_task_id` の詳細画面表示にブラウザが応答不能になる。
- **Green 条件:** (1) タイムライン表示はページネーションまたは仮想スクロールとし、初回ロードは最新 100 件とする。(2) DB クエリに LIMIT を強制する。(3) フェーズ単位のフィルタリングを UI に設ける。
- **検証対象:** §2.2 開発履歴詳細画面

### UI-03: 承認ボタンの二重クリック

- **攻撃シナリオ:** フェーズ5完了後の最終承認ボタンを2人の管理者が同時にクリックする、または1人が素早く2回クリックする。1回目で `waiting_approval` → `succeeded` に遷移した後、2回目のクリックが既に `succeeded` の行に対して何らかの副作用（本番反映の二重実行等）を引き起こす。
- **Red 条件:** 承認操作の二重実行により、本番反映が2回実行される、またはタスク状態が不整合になる。
- **Green 条件:** (1) 承認操作は `UPDATE tasks SET status='succeeded' WHERE id=? AND status='waiting_approval'` の affected rows で冪等性を保証する。(2) affected rows = 0 の場合は「既に処理済み」として UI にフィードバックする。(3) 本番反映タスク（`promote_release`）の発行は承認操作と同一トランザクションで行い、二重発行を防ぐ。
- **検証対象:** §2.2 承認機能、§1.4

## 10. ファイル運用プロトコルの破壊

### FP-01: 差分マニフェストの欠落

- **攻撃シナリオ:** 作業 AI がファイル編集前に `patches/` への差分マニフェスト出力を省略する（実装バグまたは例外による中断）。編集が失敗した場合、差分単位の巻き戻しが不可能になり、ファイルが中間状態で残る。
- **Red 条件:** 差分マニフェストなしで編集が実行され、失敗時の復旧ができない。
- **Green 条件:** (1) ファイル書き込み操作は必ず差分マニフェストの書き込み成功を前提条件とする。マニフェスト書き込みに失敗した場合は編集自体を開始しない。(2) 編集開始からマニフェスト書き込みまでの区間では、対象ファイルへの書き込みを物理的に行わない（メモリ上で差分を構築し、マニフェスト記録後に一括適用する）。
- **検証対象:** §5.4

### FP-02: 並行編集の管理 AI 制御不全

- **攻撃シナリオ:** 管理 AI が同一ファイルに対する作業計画を分離したにもかかわらず、2つの作業 AI タスクが異なる `task_id` で同一ファイルの異なるシンボルを編集する。両タスクが同時に `patches/` にマニフェストを出力し、先行タスクの適用後に後続タスクのベースが変わり、差分適用が失敗する。
- **Red 条件:** 2つのタスクが同一ファイルをそれぞれ正常に編集完了するが、最終的なファイル内容が一方のタスクの変更のみを反映している。
- **Green 条件:** (1) 管理 AI は作業計画作成時にファイル単位のロック表を `tasks.payload_json` に含め、同一ファイルに対する並行タスク発行を禁止する。(2) 作業 AI は編集開始時に対象ファイルのハッシュを記録し、書き込み前にハッシュが変わっていれば `blocked` にして管理 AI に再計画を要求する。
- **検証対象:** §5.3、§5.4

### FP-03: AI マーカーの入れ子・衝突

- **攻撃シナリオ:** タスク A が AI マーカー領域を作成し、その領域内にタスク B が別の AI マーカー領域を作成する。タスク A が完了時にマーカー領域を再生成すると、タスク B のマーカーとコンテンツが消滅する。
- **Red 条件:** AI マーカー領域の再生成が別タスクのマーカー領域を破壊する。
- **Green 条件:** (1) AI マーカー領域の入れ子を禁止する。マーカー挿入前に、挿入位置が既存マーカー領域内でないことを検証する。(2) マーカー領域の再生成前に、領域内に他タスクのマーカーが含まれていないかを検査し、含まれている場合は `blocked` にする。
- **検証対象:** §5.3

## 11. コマンドインジェクション

フェーズ1設計 §8 は「テンプレート化された引数だけを許可し、自由文字列連結を禁止する」と定めているが、これが守られない場合の検証。

### CI-01: Copilot CLI 引数への注入

- **攻撃シナリオ:** `model_contract_json.cli_profile` にシェルメタ文字（`; rm -rf /`、`` `command` ``、`$(...)` 等）が含まれる。CLI 起動時にこの値がシェル経由でサブプロセスに渡される場合、OS コマンドインジェクションが成立する。
- **Red 条件:** `cli_profile` にシェルメタ文字を含む文字列を設定した場合に、それがシェルコマンドとして実行される。
- **Green 条件:** (1) すべての CLI 実行は `subprocess.run` の配列形式（`shell=False`）で行い、シェル解釈を経由しない。(2) `cli_profile` の値は英数字、ハイフン、アンダースコアのみ許可するバリデーションを適用する。(3) バリデーション不合格時はタスクを `blocked` にする。
- **検証対象:** §1.5、§8 安全性

### CI-02: workspace_path 経由の注入

- **攻撃シナリオ:** `tasks.workspace_path` に改行文字やヌル文字を含むパスが格納される。ファイル操作やシェルコマンドの引数にこの値が使われた場合、パス解釈が予期せぬ動作をする。
- **Red 条件:** 改行やヌル文字を含む `workspace_path` がファイルシステム操作に渡され、意図しない動作をする。
- **Green 条件:** (1) `workspace_path` の書き込み時に、制御文字（0x00-0x1F、0x7F）を禁止するバリデーションを適用する。(2) 読み取り時にも同じバリデーションを再適用する。
- **検証対象:** §1.1、§4.1

### CI-03: Docker CLI ラッパーのフラグインジェクション

- **攻撃シナリオ:** 作業 AI が Docker CLI ラッパーに渡す引数として、コンテナ名やイメージ名の位置に `--privileged` を埋め込む。ラッパーが位置引数のバリデーションのみを行い、フラグ位置のパースが甘い場合、特権コンテナが起動する。
- **Red 条件:** Docker CLI ラッパーの引数パースを迂回して禁止フラグが有効化される。
- **Green 条件:** (1) Docker CLI ラッパーは固定テンプレート方式とし、引数を位置ではなく名前付きパラメータで受け取る。(2) パラメータ値にハイフン開始の文字列が含まれる場合は拒否する。(3) 最終的な Docker コマンド文字列をアセンブル後、禁止フラグの正規表現マッチを実行してから起動する。
- **検証対象:** §4.4、§8 安全性

## 12. Guardian 自体の破壊

### GD-01: Guardian の単一障害点化

- **攻撃シナリオ:** Guardian コンテナが OOM Kill またはデッドロックで停止する。Guardian 自身を監視する主体が存在しないため、停止が無期限に継続する。その間にゾンビタスクが蓄積し、再起動要求も処理されず、システムが緩やかに劣化する。
- **Red 条件:** Guardian 停止中に、ゾンビタスク検知、再起動実行、本番反映の全機能が不在になる。
- **Green 条件:** (1) ホストの systemd / launchd で Guardian コンテナに `restart: always` 相当のポリシーを設定する。(2) Dashboard が Guardian のヘルスチェックも DB 経由で監視し、異常時に人間へ通知する（Dashboard は Guardian 経由でなく直接監視する）。(3) Guardian のヘルスチェックは Docker のヘルスチェック機構で行い、Docker デーモンレベルで再起動を保証する。
- **検証対象:** §8 可用性

### GD-02: Guardian による誤った本番反映

- **攻撃シナリオ:** `promote_release` タスクの `payload_json` に不正なブランチ名やコミットハッシュが含まれている場合、Guardian が誤ったコードを本番環境に反映する。
- **Red 条件:** `promote_release` の入力検証が欠如し、未テストのコミットが本番反映される。
- **Green 条件:** (1) `promote_release` タスクの受理条件として、対象コミットのフェーズ3-5 の全テスト成功記録が `tasks` テーブル上に存在することを Guardian が検証する。(2) 対象ブランチの HEAD コミットと `promote_release` 指定コミットの一致を確認する。(3) 不一致時はタスクを `blocked` にして人間承認を要求する。
- **検証対象:** §4.5、§8 安全性

## 13. 設計横断の複合故障シナリオ

### CX-01: 自己改修 + ディスク枯渇 + ゾンビタスク の三重障害

- **攻撃シナリオ:** 自己改修タスクがフェーズ4の大規模テストで `/workspace/` に大量のテスト成果物を生成し、ディスクが枯渇する。テスト中のタスクは DB への書き込みに失敗して停滞し、ハートビートも更新できず、ゾンビ判定される。Guardian がゾンビを再取得して再実行するが、ディスク枯渇が解消されていないため再び失敗する。このループが `max_retry` まで繰り返される。
- **Red 条件:** ディスク枯渇 → DB 書き込み失敗 → ゾンビ判定 → 再実行 → 再失敗のループが `max_retry` 回繰り返され、根本原因（ディスク）が解決されないまま最終的に `failed` になる。
- **Green 条件:** (1) タスク失敗時の原因分類にリソース枯渇カテゴリを設ける。(2) ディスク枯渇起因の失敗を検知した場合、再試行前に Guardian がディスク空間の確保（古いワークスペースの削除）を試みる。(3) 空間確保に失敗した場合は再試行せずに `blocked` とし、人間介入を要求する。
- **検証対象:** §4.1、§8 可用性、EX-06

### CX-02: MCP 障害 + 秘密スキャン偽陽性 の複合

- **攻撃シナリオ:** GitHub MCP が停止中にナレッジ検索が部分結果のみを返す。作業 AI がこの不完全な情報で実装を行い、ライブラリのバージョンハッシュ（正当な高エントロピー文字列）をハードコードする。プレコミットスキャンがエントロピー閾値で偽陽性を出し、コミットがブロックされる。ブロックを解消するためにハッシュを変数に外出しすると、変数名に `HASH` が含まれ再び文脈検査に引っかかる。
- **Red 条件:** 外部障害の余波で生じた偽陽性をAIが自力で解消できず、タスクが停滞する。
- **Green 条件:** (1) プレコミットスキャンの偽陽性解消フローを定義する。同一検出ルールで連続 2 回ブロックされた場合、作業 AI は `messages` に検出内容を添えて管理 AI に例外審査を要求する。(2) 管理 AI は検出ルール ID と対象文字列の用途を評価し、ホワイトリストエントリを追加するか人間判断に委ねる。
- **検証対象:** §6.2、EX-03、SS-07

### CX-03: Blue-green 切替中の DB 接続プール枯渇

- **攻撃シナリオ:** 自己改修の blue-green 切替中に、新旧両系の Brain が同時に稼働する。両系が DB 接続プールを消費し、MariaDB の `max_connections` に到達する。Librarian や Dashboard の新規 DB 接続が拒否され、全機能が停止する。
- **Red 条件:** 切替中の二重稼働により DB 接続が枯渇し、関係のないサービスが巻き添えで停止する。
- **Green 条件:** (1) 各サービスの接続プール上限を MariaDB の `max_connections` の合計が超えないよう設定する。(2) blue-green 切替中の旧系は新規タスク取得を停止し、接続プールの縮退を行う。(3) Guardian が切替開始時に DB 接続数を監視し、閾値を超える場合は切替を延期する。
- **検証対象:** §4.5、EX-01

## 15. フェーズ2 結論

本破壊的テスト設計は、フェーズ1基本設計に対して 45 件のテスト項目を定義した。

設計から発見された主な構造的脆弱性は以下の 5 点である。

1. **Guardian の再帰的自己更新に対する手順未定義。** 聖域である Guardian 自身の更新シナリオが §4.5 の対象外であり、監視者不在の時間帯が発生しうる。
2. **隔離境界の強制機構の欠如。** `/workspace/{task_id}/` の境界は論理的なパス規約であり、シンボリックリンク、パストラバーサル、Docker マウントによる物理的な脱出を防止する機構が定義されていない。
3. **機微情報スキャンの「一段限定」ルール。** 多段エンコード、ホモグリフ、ファイル分割、バイナリ埋め込みによる迂回経路が現設計では防御されない。
4. **タスク状態遷移の形式的定義の不在。** 許可される状態遷移のホワイトリストが明示されておらず、逆行や二重リースを DB レベルで防止できない。
5. **通信ハブ（MariaDB）の全損時の退避機構の不在。** 全サービスが DB 依存であり、DB 停止時のローカル退避とリカバリの設計が記載されていない。

これらの脆弱性はフェーズ3のテスト実装（Red 確認）を経て、フェーズ4の実装で解消される必要がある。

以上をフェーズ2の破壊的テスト設計確定版とする。
